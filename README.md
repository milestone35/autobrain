# cc-autopilot

A Claude Code-native system that, for every prompt, works out the **best way to do the task** —
which skill, agent, MCP server, slash command, or built-in tool to use — selects it automatically,
and installs it when needed. It continuously scrapes the official Claude marketplace plus community
sources (GitHub, npm, the MCP registry, PyPI) into a unified **capability map** and keeps it current.

> Status: local, single-machine (no remote). Sub-projects SP1–SP11 + SP13 complete (SP12 deferred);
> the four web-discovery sources are live and the plugin ships an embedded capability map. **228
> tests** pass (indexer 109 + plugin 119).

---

## Installation

cc-autopilot is a Claude Code plugin, distributed via this repository's marketplace.

```bash
# Add this repo as a plugin marketplace (replace <github-user> with the repo owner)
/plugin marketplace add <github-user>/cc-autopilot
# Install the plugin
/plugin install cc-autopilot@cc-autopilot
```

Once installed, each prompt receives passive capability hints. Use `/route <request>` to run the
multi-agent council + executor, and `/autopilot-tune` to check the current project for optimization
gaps (CLAUDE.md, permission allowlist, hooks).

## Architecture

Two cooperating parts that share one contract — the `capability-map.json` file. Single runtime:
Node.js ≥ 18, **zero runtime dependencies**, tests via the built-in `node --test`.

```
indexer/  ── heavy scanning ──▶  capability-map.json  ──▶  plugin/  ── lightweight per-prompt router
(build the map)                  (the contract)             (consume the map in Claude Code)
```

### `indexer/` — builds the capability map

A source-agnostic pipeline: **collect → normalize → dedupe → trust → store**. Each source is an
isolated module implementing a `{ name, collect(ctx) }` contract; network access is injected
(`ctx.fetchJson`) so tests never hit the network and every pure helper is deterministic.

Sources (in authority order):

| Source | Kind | Discovers |
|--------|------|-----------|
| `official-catalog` | local | the official Claude plugin marketplace cache → trusted |
| `known-marketplaces` | local | locally-known marketplace manifests |
| `builtin-catalog` | local | Claude Code built-ins (tools, slash commands, agents, shell) — no install |
| `github` | web | repos with `.claude-plugin/marketplace.json` (code search; needs `GITHUB_TOKEN`) |
| `npm` | web | MCP server packages (`keywords:mcp` search) |
| `mcp-registry` | web | the official MCP Registry (`registry.modelcontextprotocol.io`) |
| `pypi` | web | curated seed list of MCP PyPI packages, enriched via the official JSON API |

**Trust tiers:** `builtin` (use directly), `trusted` (official or allow-listed → silent install),
`candidate` (has a repo/install but unverified → one approval), `unknown`.

See [`indexer/README.md`](indexer/README.md) for source details.

### `plugin/` — per-prompt router (the Claude Code plugin)

- **Passive hint (`UserPromptSubmit` hook):** on every prompt, loads the map, runs a deterministic
  lexical matcher, and injects the top candidate capabilities as additional context. Fail-open —
  never blocks a prompt.
- **Active flow (`/cc-autopilot:route` command + `capability-router` skill):** a multi-agent council
  (Planner + Critic, ≤ 2 rounds) reaches a validated decision, installs what's needed, then carries
  out the task. Read-only steps run automatically; side-effecting steps (shell/ssh, file writes,
  installs, MCP calls) require a single approval showing the exact command.

See [`plugin/README.md`](plugin/README.md) for the router internals.

## The capability map (contract)

`indexer/data/capability-map.json` (git-ignored; regenerated by a scan):

```jsonc
{
  "schemaVersion": 1,
  "generatedAt": "<ISO timestamp>",
  "sources": { "<name>": { "ok": true, "count": 0, "error": null, "lastRun": "..." } },
  "capabilities": [
    {
      "id": "marketplace::plugin::kind::component",
      "kind": "skill | agent | mcp | command | plugin | bang | builtin-tool | slash | builtin-agent",
      "name": "...", "description": "...", "keywords": ["..."],
      "source": { "marketplace": "...", "repo": "github:owner/repo|null", "discoveredVia": "<source>" },
      "install": { "method": "plugin|mcp", "command": "...", "package": "...|null" },
      "trust": "builtin|trusted|candidate|unknown",
      "cost": { "always_on": 0, "on_invoke": 0 }, "popularity": {}, "lastSeen": "..."
    }
  ]
}
```

When the same capability is found by multiple sources it is merged, with the more authoritative
source winning (e.g. `mcp-registry` supersedes `npm` for the same package; ecosystem-aware so an
npm package and a PyPI package that share a bare name are **not** merged).

## Build the map

> **Node note:** the indexer and the plugin's hooks call `node`. Make sure `node` is on your PATH.
> If Node lives in a conda env (not on PATH), add that env's directory to PATH (or, in a shell,
> prepend it) before running these commands.

```bash
cd indexer
npm test          # run the indexer test suite
npm run scan      # scan all sources -> data/capability-map.json
npm run status    # print a summary (counts by kind / trust / source)
```

`GITHUB_TOKEN` is optional. Without it the `github` source fails soft (contributes 0); with it,
GitHub marketplace plugins are discovered. The other web sources need no auth.

## Try it in Claude Code (local, in-place)

The plugin reads the map via a path relative to its own directory, so load it **in place** (the
dev-mode `--plugin-dir` flag does not copy the plugin, which keeps that relative path working):

```bash
# 1. Ensure `node` is on PATH (see the Node note above), then in a fresh terminal:
node --version

# 2. Launch Claude Code with the plugin loaded in place:
claude --plugin-dir "<repo>/plugin"
```

Then, inside that session:

- `/plugin list` — confirm `cc-autopilot` is loaded.
- Type any request (e.g. *"audit my api security"*) — the passive hook injects candidate hints.
- `/cc-autopilot:route audit my api security` — run the council: it decides
  (`use_existing` / `install_then_use` / `no_capability_needed`), installs (trusted silently;
  candidate/unknown with one approval), and executes the task (read-only steps auto;
  side-effecting steps with one approval).

The plugin ships with an **embedded** capability map at `plugin/data/capability-map.json`, and
`mapSource` defaults to the plugin-relative `./data/capability-map.json`. Because the path resolves
against the plugin root at runtime, this works for marketplace installs too (the plugin is copied
to a cache and the relative path resolves against that copy). See **Installation** above for the
marketplace commands.

## Configuration

`plugin/config/autopilot.config.json`:

| Key | Default | Meaning |
|-----|---------|---------|
| `enabled` | `true` | master switch for the router |
| `mapSource` | `./data/capability-map.json` | embedded map path (relative to the plugin, or absolute) |
| `topN` | `5` | max candidates the hook injects |
| `scoreFloor` | `0` | minimum lexical score to surface (0 = any signal) |
| `staleDays` | `14` | age after which a "re-scan" note is appended |
| `confidenceThreshold` | `0.6` | below this, a council decision falls back to `no_capability_needed` |
| `autoInstall` | `true` | trusted capabilities install silently; `false` shows the command without running |

The indexer's auto-trusted repos live in `indexer/config/trusted-sources.json`; the curated PyPI
seed list in `indexer/config/pypi-seeds.json`.
